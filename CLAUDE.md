# AudioBookPlayer — контекст проекта для Claude Code

## Что это
Android-приложение для прослушивания аудиокниг: MP3 (главы = отдельные файлы),
таймер сна, закладки. Дизайн плеера — в духе Яндекс Книг (тёмная тема, крупная
квадратная обложка, тонкий жёлтый прогресс-бар, капсулы для скорости/таймера).

## Стек и архитектура
- Kotlin, MVVM, minSdk 24, compileSdk/targetSdk 34.
- **Media3** (`androidx.media3`) для воспроизведения — НЕ устаревший
  `com.google.android.exoplayer`. `player/PlaybackService.kt` — это
  `MediaSessionService`, который сам создаёт уведомление, отдаёт управление
  на экран блокировки/наушники. UI подключается к нему через `MediaController`,
  а не напрямую (сервис нельзя передать в конструктор ViewModel).
- **Room + Flow** — экран библиотеки обновляется реактивно, без ручного рефреша.
- Метаданные MP3 (название/автор/длительность) читаются через штатный
  `MediaMetadataRetriever` — без сторонних библиотек для тегов.
- Добавление книг — два способа (`data/scanner/LibraryScanner.kt`):
  ручной выбор файлов (`GetMultipleContents`) и сканирование папки
  (`OpenDocumentTree`), все аудиофайлы в папке = главы одной книги.
- **Без Hilt.** DI — вручную через `utils/ViewModelFactory.kt`. Так и оставляем,
  пока проект не вырастет настолько, что это станет проблемой.
- Книга = `Book` (метаданные) + список `Chapter` (по одному MP3-файлу на главу),
  связаны через `bookId`.

## Структура
```
app/src/main/java/com/example/audiobookplayer/
├── data/
│   ├── model/        Book, Chapter, Bookmark (Room entities)
│   ├── database/      AppDatabase + DAO (BookDao, ChapterDao, BookmarkDao)

│   └── scanner/        LibraryScanner — чтение файлов/папок и метаданных
├── player/
│   ├── PlaybackService.kt   MediaSessionService — ядро воспроизведения
│   └── SleepTimer.kt        таймер сна с "продлить на N минут"
├── ui/
│   ├── library/        LibraryActivity, BooksAdapter — список книг
│   └── player/         PlayerActivity, PlayerViewModel — экран плеера
└── utils/ViewModelFactory.kt
```

## Что уже готово (MVP)
- Плеер: play/pause, перемотка на 15/30 сек, слайдер прогресса, скорость воспроизведения.
- Таймер сна с пресетами (5/15/30/60 мин).
- Закладки: добавление на текущей позиции, хранение в БД.
- Библиотека: список книг, добавление файлами или папкой.

## Правила работы над проектом
- Не возвращать `com.google.android.exoplayer` — только `androidx.media3`.
- Не добавлять сторонние библиотеки для чтения тегов — `MediaMetadataRetriever` хватает.
- Списки из БД — только через `Flow`, не через `suspend fun getAll()`.
- Не подключать Hilt без явной просьбы.
- Дизайн — тёмная тема, стиль Яндекс Книг (минимализм, крупная обложка, тонкие
  элементы управления). Плюс покерная тема (владелец — покерист): приложение
  называется **All-In Audio**, акцент — не цвет, а тонкие чёрно-белые игральные
  масти (♠ ♣ ♦ ♥) как элемент управления/декор, а не цветастый вегас-стиль.
  Сейчас использовано: ♠ — закладки, ♣ — таймер сна, ♦ — скорость воспроизведения.
  ВАЖНО: суды рендерятся через Unicode text-presentation selector (U+FE0E) после
  символа — без него Android рисует их цветными эмодзи вместо монохромного текста.
  Иконка приложения — `drawable/ic_launcher_poker.xml` (пиковая масть на чёрном).
  Splash-экран — `drawable/ic_splash_spade.xml` + `Theme.AudioBookPlayer.Splash`.
  Не скатывать это в кислотный "казино"-стиль — держать минимализм.

- **СЛЕДУЮЩАЯ ЗАДАЧА (пожелание владельца от 09.07.2026):** ~~заменить системные
  плейсхолдеры на авторские иконки в 3D-стиле~~ — ГОТОВО (10.07.2026). Создан набор
  `drawable/ic_*_3d.xml` (play, pause, rewind, forward, close, bookmarks_list, delete,

  VectorDrawable — это поддерживается только с API 24+ (у нас как раз minSdk 24, так что
  ок, но если minSdk когда-нибудь понизят — эти иконки сломаются на старых устройствах).

## Известные и исправленные баги
- **10.07.2026 — плеер не играл звук, кнопки не реагировали.** Причина: гонка состояний
  в `PlayerViewModel.loadBook()` — `controller?.setMediaItems(...)` мог выполниться до
  того, как `MediaController` успевал асинхронно подключиться к `PlaybackService`, и
  `?.` молча пропускал вызов, поэтому файлы книги никогда не загружались в плеер.
  Исправлено через `CompletableDeferred<MediaController>` (`controllerReady`) — теперь
  `loadBook()` дожидается реального подключения перед `setMediaItems/prepare/play`.
  Если снова появятся симптомы "тихо, кнопки не реагируют" — первым делом проверять
  именно эту гонку, а не аудиофайлы/права доступа.

- **10.07.2026 — обратный отсчёт таймера сна.** Капсула ♣ в плеере теперь живая:
  пока таймер активен, показывает "♣ MM:SS" (обновляется раз в секунду через
  `PlayerViewModel.sleepSecondsLeft`), при завершении/отмене возвращается к
  "♣ Таймер сна". Реализовано в `PlayerActivity.observeState()`.

- **10.07.2026 (2) — удаление книг.** Раньше такой функции не было вообще. Добавлено:
  иконка корзины (`ic_delete_3d`) в каждом элементе списка библиотеки → диалог
  подтверждения → `BookRepository.deleteBook()`, который теперь каскадно чистит
  главы и закладки этой книги (`ChapterDao.deleteChaptersForBook`,
  `BookmarkDao.deleteBookmarksForBook`), а не только саму запись книги — иначе в БД
  оставался бы мусор. Сами аудиофайлы на телефоне не трогаются, удаляется только
  запись в библиотеке приложения.

- **10.07.2026 (3) — плеер "опять не работал".** Оказалось, что предыдущий фикс
  (`controllerReady`) закрыл гонку только в `loadBook()`, но `togglePlayPause`,
  `seekTo`, `seekToBookmark`, `skipForward/Back`, `setSpeed`, `addBookmark` всё ещё
  использовали `controller?.let{}` — то есть молча ничего не делали, если пользователь
  нажимал кнопку до завершения подключения MediaController (это происходит очень
  быстро, но не мгновенно). Исправлено единым способом: добавлен приватный хелпер
  `withController { action -> ... }`, который дожидается `controllerReady.await()`
  перед ЛЮБЫМ действием с плеером — теперь ни один вызов не может потеряться из-за
  гонки. Если проблема "плеер не реагирует" всплывёт снова — искать `controller?.`
  без `withController` по всему `PlayerViewModel.kt`, такого быть не должно.

- **11.07.2026 — плавное затухание таймера сна.** Последние 10 секунд перед
  срабатыванием таймера громкость плавно снижается с 1.0 до 0.0 (ступенчато,
  раз в секунду через `PlayerViewModel.applyFadeVolume`, используя `MediaController.
  volume` — он реализует `Player`, так что регулировать громкость можно прямо из
  ViewModel, без похода в PlaybackService). По завершении — пауза + громкость
  сбрасывается обратно на 1.0 для следующего раза. Volume также принудительно
  сбрасывается на 1.0 при старте нового таймера и при отмене — на случай, если
  предыдущее затухание было прервано на середине.
